App — a real app built on CloudflareShard
Everything below is the SAME live cluster the other rooms watch — a real multi-tenant view over the TPC-C demo data, read through /v1/table-scan, for one of three controlled demo tenants (warehouse 1, 2, or 3). Switching tenant re-queries live. Nothing here is mocked.
Tenant
Pick a demo warehouse — each one is a separate CloudflareShard tenant, isolated by tenantId at the routing layer, not by anything this app's code does.
Customers —
The first 5 customer rows for this tenant, via /v1/table-scan on tpcc_customer — no arbitrary filter, just this tenant's own rows.
Inventory —
The first 5 stock rows for this tenant, via /v1/table-scan on tpcc_stock. The Restock action below writes to these same rows.
Restock — a same-tenant transaction
One button, one real /v1/tx call: a same-tenant, multi-row transaction across the first 2–3 of the stock rows shown above (well under /v1/tx's real 8-participant cap) — every row in the transaction updates, or none does. This is not a full TPC-C order and not a cross-tenant transaction; it is exactly what it looks like: one small, real, bounded two-phase-commit transaction, guarded by an optimistic-concurrency where clause on every row so a concurrent writer can't be silently overwritten.
How little code this takes
The two calls below are copy-accurate to CloudflareShard's real /v1/table-scan and /v1/tx API (see the CloudflareShard project's root README) — and they are this screen's entire data layer. No ORM, no manual shard routing, no migration coordinator: sharding, secondary indexes, and the 2PC transaction are the platform's job, not this app's.

        Playground — developer primitive console
      


      

        

          Every panel below fires a real request against the live cluster, scoped to a controlled demo tenant
          (warehouse 1, 2, or 3) over the
          same 8 TPC-C demo tables the load engine already writes. Dropdowns only offer values the server actually
          accepts — an invalid request can't be built here, only rejected honestly by the cluster.
        


        


          
          

            

              Mutate
            

            

              A single write via /v1/mutate — insert / update / delete / upsert one row by
              partition key. Every mutate carries a requestId; replaying the exact same
              request with the same id returns the identical cached result (no double effect), while replaying the
              same id with a different body is refused with a 409 — see the idempotency demo below the form.
            

            

              
              
              
              
              
              
              
              
              
              
              
              
              
              
              

              

                
                
              

            

            

            

          


          
          

            

              Tx
            

            

              Same-tenant two-phase commit via /v1/tx — 1 to 10 mutations against the
              chosen warehouse, applied atomically: every mutation commits, or none does.
            

            

              
              
              

              
              
              
              

              
            

            

          


          
          

            

              Index Query
            

            

              A point lookup via /v1/index-query against one of the table's registered
              secondary indexes — the same indexes the real TPC-C load mix already queries.
            

            

              
              
              
              
              
              
              
              
              
              
              

              
            

            

          


          
          

            

              Table Scan
            

            

              A paginated, tenant-scoped full scan via /v1/table-scan. Pass the
              cursor from a result back in to page forward.
            

            

              
              
              
              
              
              
              
              
              
            

            

          


          
          

            

              Routing Inspector
            

            

              Two-step hash routing, exactly as the cluster applies it: tenantId picks a
              catalog shard (hashKey(tenantId) % catalogShardCount), then
              tenantId:table:partitionKey picks a vBucket within that catalog
              (hashKey(...) % totalVBuckets), which the live vBucket map resolves to an
              owning physical shard. Read-only — reflects the cluster's real current state, including a reshard
              happening in another room right now.
            

            

              
              
              
              
              
              
              

                
                
              

            

            

            

          


          
          

            

              SQL
              Operator-only · read-only
            

            

              Runs a single read-only SELECT/EXPLAIN statement
              against one shard via /v1/sql, server-side under the admin token — the
              browser never sees it. Writes, DDL, and multi-statement payloads are rejected with a 400 before they
              ever reach the cluster.
            

            

              
              
              
              
              
              
              
              
              
              
              

              
            

            

          


          
          

            

              Scatter
              Operator-only · read-only
            

            

              Fans a single-table, no-JOIN SELECT across every tenant on every shard via
              /v1/scatter, server-side under the admin token. Scoped to one of the demo
              tables below — no arbitrary or joined table is accepted.
