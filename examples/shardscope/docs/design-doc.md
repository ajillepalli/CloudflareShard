# Design: Shardscope — a live "mission control" demo for CloudflareShard

Generated by /office-hours on 2026-07-16
Branch: feature/tpcc-benchmark
Repo: ajillepalli/CloudflareShard
Status: DRAFT
Mode: Builder

## Problem Statement

CloudflareShard is a genuinely capable piece of infrastructure — a multi-tenant
sharded SQL layer on Cloudflare Durable Objects with distributed 2PC transactions,
secondary indexes, scatter reads, table scans, idempotent mutations, and (the crown
jewel) online resharding with a fenced, checksum-verified cutover: vBucket split,
vBucket migrate, and full shard drain, all without downtime.

None of that is visible. The two existing artifacts under `examples/` prove the
plumbing works — `rpc-consumer` shows the RPC/service-binding data path, and
`tpc-c-benchmark` drives a realistic OLTP mix — but neither lets a human *see*
anything. A potential user evaluating this as their datastore, or a developer
deciding whether to build on it, has nothing to look at. The most impressive and
hardest-to-believe claim (reshard a live cluster under load, lose zero writes)
is exactly the one with no demo.

We need one functional, full-stack demo product that makes the invisible visible
and exercises every primitive under real traffic.

## What Makes This Cool

The "whoa" is a single screen where a viewer watches a live, multi-tenant SQL
cluster reshard itself under continuous OLTP load and never drops a write:

- A **hot shard turns red** as the TPC-C load generator hammers it.
- An operator (or a scripted scenario) hits **Split** / **Migrate** / **Drain**.
- The **vBucket map animates** — you literally watch buckets move from one shard
  to another, with the fenced dual-write backfill and checksum cutover happening
  live.
- A persistent scoreboard reads **`writes: N · lost: 0 · checksum OK`** the entire
  time. The heat rebalances. Nobody paused traffic.

For a database, that is the emotional core of the whole product: the power is
systemic and normally invisible, and this makes it legible in one glance. The
second delight is the DX flip-side — a Playground room where a developer runs a
2PC transaction across shards, creates a secondary index, watches which physical
shard a `tenant:table:key` routes to, and realizes the app they'd build gets all
of this for free.

## Constraints

- **Runtime:** Cloudflare Workers + Durable Objects. The demo is one Worker plus a
  static frontend, deployed with the same `wrangler` pipeline the project already
  uses. No new infra category.
- **Data source is the real admin API, not a mock.** `/admin/status` returns
  `{ initialized, catalogShardCount, shards: {total, active, draining}, catalogs[] }`;
  `/admin/shard-stats` fans out per-shard; there are status endpoints for migrate
  (`-status` + `-abort`), drain (`-status` only), and topology-lock. Per-tenant
  routing is a pure function (`hashKey(tenant:table:pk) % totalVBuckets` → map
  lookup) that the frontend can reproduce exactly (`src/hash.ts`).
- **The vBucket→shard map is NOT yet gateway-exposed — this is net-new engine work,
  not a free dependency.** `handleVbucketMap` (`src/catalog.ts:1580`) is a
  CatalogDO-internal, DO-binding-only route; it is absent from the Worker `ROUTES`
  table and from the `CloudflareShardRpc` entrypoint. The whole headline feature
  (animated vBucket map, Topology room, Playground routing inspector) depends on it.
  Adding `/admin/vbucket-map` (a gateway route + RPC method that fans out to each
  catalog's internal `/vbucket-map` and merges) is a **required first work item**,
  and it must also return per-vbucket `migration_status` / `target_shard_id` so
  buckets-in-motion can be animated (today it returns only `{vbucket, shardId}`).
- **Auth is two-tier, but NOT uniformly `/v1/*`.** Corrected by eng review:
  `/v1/mutate`, `/v1/tx`, `/v1/index-query`, `/v1/table-scan` are per-tenant
  (tenant token). But `/v1/sql` and `/v1/scatter` are **operator/admin-gated**
  (`scatterCore` → `requireAdminAuthFromHeader`, `src/index.ts:3627`) because they
  read across tenants indiscriminately. So the Playground must present `sql`/`scatter`
  as operator tools, not tenant-data-plane primitives, or it's a confused-deputy
  admin SQL console dressed as a demo. The Shardscope Worker proxies data calls so
  the browser never holds raw credentials; the proxy is a required piece. See the
  eng-review addendum for tenant-token durability + Shardscope's own auth gate.
- **Reuse, don't rebuild, the workload.** `examples/tpc-c-benchmark/` already drives
  all 5 TPC-C transaction types across 9 tables, one tenant per warehouse, over
  `/v1/mutate` + `/v1/tx` + `/v1/index-query` + `/v1/table-scan`. It becomes the
  live load source so resharding happens under real traffic and the correctness
  meter has something real to check.
- **Auth model:** gated behind the existing admin token. This is a dev/demo/ops
  tool, not a hardened public multi-user service. Say so plainly in the README.
- **All admin mutations already serialize** through the durable topology-operation
  lock, so the console can't let an operator start two conflicting reshapes at once —
  the UI surfaces `/admin/topology-lock-status` instead of inventing its own guard.

## Premises

1. The demo is ONE Worker + static frontend "mission control" with four rooms —
   **App**, **Topology**, **Reshard console**, **Playground** — not four separate
   projects. **Agreed** (user: "why not all of the above").
2. It reuses the existing TPC-C harness as the live load source; no throwaway
   workload. **Agreed.**
3. The killer moment is visual: hot shard → live split/drain → heat rebalances,
   with a persistent `writes/lost/checksum` scoreboard proving zero downtime.
   **Agreed.**
4. It ships as a dev/demo tool, not a hardened public service. Admin controls use
   the admin token; the demo Worker proxies `/v1/*` data calls under per-tenant
   tokens it mints server-side (two-tier auth), so the browser never holds admin
   credentials. **Agreed.**
5. Build shape: the whole four-room product as one cohesive architected push,
   with the real-time layer and one design language designed up front (not a
   wow-first spike that gets decorated later). **Agreed** (user chose "Whole
   product, one push" at D2).

## Approaches Considered

### Approach A: Wow-first spike, then expand (minimal viable)
Ship a plain dashboard that proves the reshard-under-load visualization first,
then layer the cinematic story and the polished app + playground.
- Effort: M. Risk: Low.
- Pros: de-risks the hardest thing first; shareable wow in the first iteration;
  additive phases, never a rewrite.
- Cons: interim iterations look plain; the "whoa" lands before the polish does.
- Reuses: TPC-C harness, all `/admin/*` read + reshard endpoints.
- **Not chosen** — user explicitly wanted the complete, cohesive product.

### Approach B: Whole four-room product, one architected push (ideal) — CHOSEN
Design the shared real-time layer, data model, and design system up front, then
build App + Topology + Reshard + Playground together as one coherent product.
- Effort: L. Risk: Med.
- Pros: most cohesive result, one consistent design language throughout; forces
  the full real-time/data architecture to be designed once, cleanly; strongest
  single artifact for both audiences and for the project's own docs/landing.
- Cons: longest time to first demoable wow; must be disciplined to prove the
  reshard-under-load core early inside the bigger build so polish doesn't outrun
  correctness.
- Reuses: TPC-C harness (load), `rpc-consumer` pattern (data path), all `/admin/*`
  endpoints (topology + reshard), `src/hash.ts` (client-side routing preview).

### Approach C: Cinematic chaos-theater first (creative/lateral)
A single scripted, auto-narrated scenario page built to be recorded as a 90-second
clip; interactive rooms come after.
- Effort: M. Risk: Low-Med.
- Pros: most shareable/viral artifact first; forces a tight narrative of *why*
  resharding matters.
- Cons: needs the same reshard backend as A/B anyway; front-loads story polish
  before interactive control is proven.
- **Not chosen as the frame**, but folded in — the scripted "Auto-run scenario"
  becomes a mode inside the Reshard room in Approach B, so we still get the
  shareable clip without making it the whole product.

## Recommended Approach

**Approach B — the whole four-room product, one cohesive push — with Approach C's
scripted scenario folded in as an "Auto-run" mode inside the Reshard room.**

Rationale: the user's goal is to sell both audiences at once ("users AND
developers") and they explicitly chose the complete build. A database's power is
systemic, so a partial demo undersells it; the cohesive product is the honest
representation. We keep engineering discipline by ordering the *internal* build so
the riskiest thing (reshard-under-load visualization + correctness meter) is proven
early inside the larger effort — see Next Steps. Folding C's scripted scenario in
gives a recordable 90-second story without spawning a separate artifact.

### Product shape: one Worker, four rooms

Name: **Shardscope** (working title). One new Worker (`examples/shardscope/`) that
service-binds to the existing gateway Worker, plus a self-contained static SPA
served from it. Shared top bar shows cluster identity + the live scoreboard
across all rooms.

- **Room 1 — App (DX story).** A real multi-tenant view over the TPC-C data: a
  tenant switcher (each warehouse = a tenant), an orders/customers/stock browser
  driven by `/v1/index-query` + `/v1/table-scan`, and a "New Order" / "Payment"
  action that fires a real `/v1/tx` 2PC. **Honesty constraint (eng review):** in
  this repo `/v1/tx` caps at 8 participants and is same-tenant only, and TPC-C New
  Order is *deliberately split* into fragments (`examples/tpc-c-benchmark/README.md`).
  The App room must show a real multi-row same-tenant 2PC and label it accurately —
  it must NOT imply a full TPC-C order is one cross-shard cross-tenant transaction.
  A visible "this whole screen is N lines of app code" callout with a link to the
  source. Message: build your app on this and sharding/tx/indexes come for free.
- **Room 2 — Topology.** Live map of catalog shards → physical shards → vBuckets,
  polled from `/admin/status` + the vBucket map + fanned-out `/admin/shard-stats`.
  Per-shard heat (rows / recent write rate), active vs draining state, index
  placement rings. This is the room that makes the database legible.
- **Room 3 — Reshard console.** Operator controls for `split-vbucket`,
  `migrate-vbucket`, `drain-shard`, guarded by live `/admin/topology-lock-status`.
  Progress/abort controls match what the engine actually offers: `migrate-vbucket`
  has both `-status` and `-abort`; a split is internally a migration so its progress
  is read via `migrate-vbucket-status`; `drain-shard` has `-status` only (no abort).
  Scope abort UI to migrate, or add the missing engine endpoints as explicit new
  work if drain-abort is wanted. Two modes: **Manual** (operator clicks) and
  **Auto-run scenario** (the scripted chaos-theater story). The scoreboard
  (`writes · lost · checksum OK`) is anchored here and mirrored in the top bar. Its
  correctness signal comes from a continuous background reader verifying a known key
  set across the cutover; the `checksum OK` line is *inferred from a successful,
  checksum-verified cutover* unless we add an endpoint that surfaces the engine's own
  checksum verdict (see Open Questions) — checksums today are computed and compared
  only inside the cutover (`src/catalog.ts:3117`) and a mismatch aborts the migration
  rather than reporting a readable verdict.
- **Room 4 — Playground.** Interactive console for every primitive: `/v1/sql`
  (data-only), `/v1/mutate` (with `requestId` to demo idempotency — replay the same
  request, watch it reject a mismatched body instead of returning stale), `/v1/tx`,
  `/v1/index-query`, `/v1/scatter`, `/v1/table-scan`. A routing inspector reproduces
  `hashKey(tenant:table:key) % totalVBuckets` client-side and highlights the target
  shard on the Topology map, so a developer *sees* deterministic single-shard
  routing.

### Real-time layer (designed up front, per the chosen build shape)

The Worker exposes a read-side aggregation endpoint (or SSE stream) that polls the
admin API on a short interval, computes per-shard heat + the current vBucket map +
lock state + scoreboard counters, and pushes deltas to the SPA. The frontend never
calls `/admin/*` directly for the firehose; it subscribes to this one stream. This
is the single architectural decision that has to be right on day one because all
four rooms consume it. Load generation runs as a controllable background driver
(wrapping the TPC-C harness) with start/stop/concurrency exposed in the UI.

## Open Questions

1. **Where does the load generator run?** Options: (a) in-browser fetch loop from
   the SPA, (b) a co-deployed Worker/queue driver, (c) the existing Node
   `run.mjs` harness driven out-of-band. (b) is cleanest for a hosted demo; (a) is
   simplest for local `wrangler dev`. Likely support both, default per environment.
2. **Live deploy vs local-only.** Is there a public demo URL (needs a persistent
   seeded cluster + cost/rate-limit thought), or is this "clone and `npm run dev`"?
   Affects the load-source default and whether we need a reset endpoint.
3. **Correctness meter fidelity.** Minimum viable is a continuous known-key reader
   asserting expected values across cutover; the strong version taps the existing
   checksum-verified cutover so the meter reflects the engine's own verification,
   not a parallel check. Decide how deep to wire in.
4. **Naming.** "Shardscope" is a working title; confirm or pick another before it
   leaks into paths and docs.

## Success Criteria

- A viewer can, in one screen, start load, watch a shard go hot, trigger a live
  split or drain, and see heat rebalance while `lost: 0 · checksum OK` holds — no
  paused traffic, no manual refresh.
- Every public primitive (`sql`, `mutate` incl. idempotent replay, `tx`,
  `index-query`, `scatter`, `table-scan`) is exercisable from the Playground and
  visibly reflected on the Topology map.
- The App room performs a real cross-shard 2PC transaction from a button click and
  shows the result.
- A developer can read the App room's source and see how little code sits between a
  UI action and a sharded, transactional write.
- The Auto-run scenario produces a clean ~90-second recording with no operator
  input.
- `npm run dev` + a documented seed command brings the whole thing up locally.

## Distribution Plan

- Lives in-repo at `examples/shardscope/` (mirrors `examples/rpc-consumer/` and
  `examples/tpc-c-benchmark/`), with its own README and `wrangler` config.
- Deployed via the project's existing `wrangler deploy` pipeline — no new CI
  category. If a public demo URL is wanted (Open Question 2), add a GitHub Actions
  job that deploys the demo Worker on merge to a demo environment plus a seed step.
- README includes: local quickstart, the seed command, a "record the 90-second
  scenario" note, and a labeled screenshot/GIF of the reshard moment for the repo's
  top-level README and any landing page.

operator endpoint, not a demo-only concern.

## Next Steps

Even though we're building the whole product, order the internal work so the risky
core is proven before the polish:

0. **Expose the vBucket map at the gateway** (`/admin/vbucket-map` route + RPC,
   fan-out-and-merge across catalogs, plus per-vbucket `migration_status` /

2. **Topology room** rendering that stream — static cluster first, then live heat.

3. **Reshard console (Manual)** wired to split/migrate/drain + status/abort, with

4. **Reshard console (Auto-run scenario)** — script the seed→ramp→hot→split→drain→
   heal story for the 90-second recording.
5. **Playground room** — every primitive + the client-side routing inspector +
   idempotent-replay demo.
6. **App room** — the multi-tenant TPC-C view with a real 2PC action and the
   "N lines of app code" DX callout.
7. **One design language pass** across all four rooms + README, screenshots/GIF,
   and (if chosen) the demo-deploy CI job.

## What I noticed about how you think

- You answered "why not all of the above?" to a four-way either/or. You didn't
  hedge or ask which was cheapest — you rejected the premise that a demo of a
  systemic capability should be partial. For a database that's the correct
  instinct: the power *is* the whole, so a slice undersells it.
- Then at the build-shape fork you chose "whole product, one push" over the
  de-risked "wow-first, then expand" I recommended. You're optimizing for a
  coherent finished artifact over the safe incremental path — a taste call, and a
  defensible one here, since the cohesion is the point of the demo.
- Both choices point the same direction: you want the complete thing, built once,
  built right. That's the same value the project's own spec shows — real fenced
  cutovers, real checksums, no demo-path shortcuts.

# Engineering Review Addendum (plan-eng-review, 2026-07-16)

## Decisions locked this review
- **Load source (Issue 1):** BOTH a Node local driver (reuse `run.mjs`) and a
  Worker-native self-driving load engine (port the TPC-C tx *mix*, drive via DO
  alarms / rolling subrequest batches sized to Workers CPU+subrequest caps),
  pluggable behind one start/stop/concurrency control. Hosted demo must sustain
  load with no laptop attached.
- **Aggregator (Issue 2):** ONE shared singleton poller (a Durable Object with an
  alarm) polls the admin API once per tick regardless of tab count and fans one
  snapshot out to all SSE subscribers. Control-plane read load stays O(1) in
  viewers so the dashboard never degrades the reshard it films.
- **Scope (cross-model tension):** Keep whole-product (all four rooms), but the
  reshard-core proof is a HARD PHASE-1 GATE (see Sequencing gate). Codex's risk
  folded in as ordering, not a scope cut.

## Corrections folded from the outside voice (code is authoritative)
1. `/v1/sql` + `/v1/scatter` are ADMIN-gated, not tenant-plane. Playground labels
   them operator tools; never proxy them under a browser-reachable path without a
   Shardscope-side gate.
2. Routing is two-level: `catalog-${hashKey(tenantId) % CATALOG_SHARD_COUNT}`
   (`src/index.ts:106`) then vBucket within that catalog. The routing inspector and
   the map merge must be catalog-aware and key by `{catalogShardId, vbucket}` —
   vBucket IDs are catalog-local.
3. Tenant tokens are returned only at registration and stored as hashes. The proxy
   CANNOT hold them in isolate memory (lost on eviction/deploy) and must NOT
   re-register/rotate the warehouse tokens the Node harness cached in
   `.tpcc-tenants.json`. Store minted tokens durably (KV or the aggregator DO).
4. Shardscope needs its OWN auth gate. The demo Worker holds the admin secret;
   without a gate it's a public admin+SQL console. Add a Shardscope access gate
   (env-configured), and for any public deploy, disable topology-mutating controls
   or sandbox them to a throwaway cluster (blast-radius: a leaked path can drain a
   cluster).
5. Shard-id source: `/admin/status` returns only counts, `/admin/shard-stats` needs
   a `shardId`. The aggregator needs an authoritative shard-id list including
   draining and pre-cutover-target shards — derive it from the vBucket map
   (`shardId` + `target_shard_id` columns), not from `/admin/status`.
6. Hot-shard skew driver (Issue: "red shard isn't free"): vanilla TPC-C hashes rows
   across vBuckets and may never produce a clear hot shard. The load engine needs a
   deterministic skew mode that targets keys known (via the routing function) to
   land on one vBucket/shard. Without it there is no visible "red."
7. Correctness channel must be defined precisely, not smoke:
   - `writes`: count acknowledged (2xx) mutations; separately track retried-via-
     idempotency and expected-aborted 2PCs so they're not miscounted as "lost."
   - `lost`: a continuous reader over the ACTUAL moving-vBucket write set (not a
     fixed sample) asserting read-back equals last-acked write across cutover.
   - `checksum`: the engine checksum is a cutover EVENT, not a live invariant. Label
     it "cutover verified" (last-known), or add an endpoint that surfaces the
     cutover's checksum verdict. Do not render inference as continuous proof.
8. 90-second auto-run vs ~1m45s seed: needs a persistent pre-seeded cluster or a
   reset/snapshot mechanism so the recordable scenario starts from a hot state.

## Sequencing gate (Phase 1 must pass before Phase 2 exists)
- **Phase 1 (proof, hard gate):** `/admin/vbucket-map` exposure (catalog-aware
  merge + migration_status/target_shard_id) → shared singleton aggregator + SSE →
  Topology room live heat → skew driver → Reshard console (manual) under load →
  defined correctness channel showing `lost: 0` across a real split AND drain.
  **Gate:** a viewer sees a shard go red, a split/drain rebalance it, and the
  correctness channel hold, on the hosted URL. Only then:
- **Phase 2 (product):** App room, Playground, Auto-run cinematic, tenant proxy for
  the DX rooms, one design-language pass, README + GIF, demo-deploy CI.

## Acceptance criteria / SLO budget (was missing — add before build)
Define and assert in tests + a demo-health check:
- Admin poll frequency and max subrequests per aggregator tick.
- Max allowed OLTP throughput degradation while the dashboard is polling (e.g.
  p95 latency delta < X% vs no-dashboard baseline) — proves the dashboard doesn't
  invalidate its own demo.
- Max reshard duration under the demo's load level.
- Dashboard tab-count the singleton aggregator supports.
- Worker CPU-time / subrequest error rate under the Worker-native load engine.
- DO alarm lag tolerance for the poller.

## What already exists (reuse, don't rebuild)
- `examples/tpc-c-benchmark/` — 5 TPC-C tx types, per-tenant tokens, HTTP client.
  Reused as the Node driver AND the source of tx logic to port into the Worker
  engine. Its `.tpcc-tenants.json` token cache constrains the proxy (correction 3).
- `examples/rpc-consumer/` — the service-binding data-path pattern the demo Worker
  follows to reach the gateway.
- `src/hash.ts` `hashKey()` + `catalogShardIdForTenant`/vBucket logic — the routing
  inspector and skew driver reuse these; do not reimplement the hash.
- `vbucket_map` migration columns (`migration_status`, `target_shard_id`,
  `cutover_started_at`, `cutover_stall_reason`, `cleanup_source_shard_id`,
  `src/catalog.ts:213-249`) — already durable; `/admin/vbucket-map` is a SELECT +
  exposure, not new bookkeeping.
- The topology-operation lock + `/admin/topology-lock-status` — the console surfaces
  it instead of inventing its own concurrency guard.

## NOT in scope (considered, deferred)
- WebSocket/bidirectional transport — SSE one-way is sufficient for a read dashboard.
- Hardened multi-user public auth / rate limiting — demo is gated, not a SaaS.
- Exposing `drain-shard-abort` / `split-vbucket-status` engine endpoints — out of
  scope unless the console needs drain-abort; scope abort UI to migrate.
- A bespoke non-TPC-C workload — TPC-C + skew driver covers the demo.
- Analytics/joins/aggregation rooms — the engine has no join engine by design.

## Failure modes (new codepaths)
- **Aggregator DO polls during cutover** → contends with migration alarms. Mitigation:
  O(1) singleton + a poll budget in acceptance criteria. Test: latency-delta assert.
- **Worker-native load engine hits CPU/subrequest cap** → silent throughput cliff
  mid-demo. Needs error-rate telemetry + visible degradation, not a silent stall.
- **Tenant token lost on isolate eviction** → App/Playground 401 mid-demo. Durable
  storage + a re-mint path that does NOT rotate harness-cached tokens. Test required.
- **Skew driver overshoots** → one shard so hot the migration can't drain it under
  load. Bounded skew + a documented safe load level.
- **Correctness reader samples non-moving keys** → false green `lost: 0`. Reader must
  track the moving vBucket's write set. This is the highest-risk silent failure —
  **critical gap** if shipped without a test that injects a lost write and sees red.

## Worktree parallelization
- **Lane A (core, gating):** `/admin/vbucket-map` (gateway/`src/`) → aggregator DO →
  Topology + Reshard rooms. Sequential, shared with `src/`.
- **Lane B (load engine):** Worker-native TPC-C mix + skew driver. Independent of
  Lane A's `src/` gateway change until wiring; can start in parallel.
- **Lane C (Phase 2, after gate):** App room, Playground, tenant proxy, cinematic,
  design pass. Depends on Lane A's SSE contract.
- Launch A + B in parallel worktrees; merge; then C. Conflict flag: A and any
  gateway auth change both touch `src/index.ts` routes — coordinate.
