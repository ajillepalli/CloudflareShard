# cloudflare-shard-tpcc-benchmark

A TPC-C-derived OLTP demo and benchmark for CloudflareShard (issue #16), adapted to this project's actual query primitives (`/v1/mutate`, `/v1/tx`, `/v1/index-query`, `/v1/table-scan`) over the existing HTTP API.

**This is not TPC-DS** (the analytics/decision-support benchmark originally suggested — CloudflareShard has no join or aggregation engine, deliberately, so it doesn't fit). **This is also not an official or certifiable TPC-C submission** — several deliberate, documented adaptations were needed to fit this project's architecture. Read "Deviations from official TPC-C" below before quoting any numbers this produces as if they were a real TPC-C result.

## What this actually demonstrates

CloudflareShard's `CoordinatorDO` 2PC and sharded `mutate` path under a standard, realistic OLTP transaction mix: 9 tables (warehouse/district/customer/order/order-line/stock/etc.), all 5 TPC-C transaction types, run at a configurable scale against a local `wrangler dev` instance or a live deployment.

## Quick start

```bash
cd examples/tpc-c-benchmark

# 1) Seed a cluster (needs /admin/init already called against the target — see the main README's quickstart step 1)
node src/run.mjs seed --base-url http://localhost:8787 --admin-token $ADMIN_TOKEN

# 2) Run the benchmark
node src/run.mjs benchmark --base-url http://localhost:8787 --duration 30
```

`seed` writes tenant tokens + cached reference data to `.tpcc-tenants.json` (gitignored — it holds plaintext bearer tokens, never commit it). `benchmark` reads that same file, so you don't need to re-seed between runs unless you want fresh data.

### `seed` options

| Flag | Default | Notes |
|---|---|---|
| `--warehouses` | 2 | One tenant per warehouse (see "Tenancy model" below) |
| `--districts-per-warehouse` | 10 | Matches the official spec |
| `--customers-per-district` | 100 | Official spec: 3000 — reduced for faster demo seeding |
| `--items` | 200 | Official spec: 100,000 — reduced for faster demo seeding, and to keep New-Order's stock-row collision rate low (see below) |
| `--tenants-file` | `.tpcc-tenants.json` | Where tenant tokens + cached reference data are written |
| `--start-warehouse` | 1 | Warehouse IDs (and their tenant IDs / row keys) are deterministic, so seeding always starting at 1 means you can't add a *fresh* batch of warehouses to a cluster that already has warehouse 1..K seeded without colliding — pick a higher starting ID instead |
| `--seed-orders` | off | Also seeds one pre-existing order (+5-15 order lines) per customer, ~30% left undelivered — matches the official spec's initial-load convention, but is by far the most expensive part of seeding (thousands of extra `order_line` rows, and a proportionally slower index backfill). Off by default so the out-of-the-box seed stays fast; New-Order transactions create real orders for Delivery to act on within seconds of a benchmark run starting anyway |

**How long seeding actually takes**: measured locally (Windows, `wrangler dev`) at the defaults above (2 warehouses, no `--seed-orders`): **~1m45s**. Each HTTP round trip to a local `wrangler dev` instance costs roughly 300-500ms (Durable Object dispatch overhead, not something this benchmark's own code controls), and seeding issues on the order of 2,800 individual `/v1/mutate` calls at that scale — the time is almost entirely round-trip count × per-call latency, not computation. Scaling `--warehouses`/`--customers-per-district`/`--items` up multiplies seed time roughly proportionally; a live deployment will typically have lower per-call latency than a local dev instance, but network round-trip time enters instead.

### `benchmark` options

| Flag | Default | Notes |
|---|---|---|
| `--duration` | *(required)* | Seconds to run |
| `--concurrency` | 10 | Concurrent workers, each looping continuously |
| `--tenants-file` | `.tpcc-tenants.json` | Must match what `seed` wrote |
| `--base-url` | *(from tenants file)* | Override if reusing a tenants file against a different host |

Each worker picks one of the 5 transaction types at random, weighted to the standard TPC-C mix (45% New-Order, 43% Payment, 4% Order-Status, 4% Delivery, 4% Stock-Level), runs it against a randomly chosen warehouse/district/customer, and records `{type, ok, latencyMs, error?}`.

## Reading the report

```
=== TPC-C-derived benchmark report (CloudflareShard, issue #16) ===
Duration: 30s
Total transactions attempted: 142
tpmC-equivalent (successful New-Order transactions / minute): 118.00

Per-transaction-type latency (ms, successful attempts only) and outcome counts:
type            n     ok    err      p50      p95      p99
payment           61    61     0    1001.7   1493.6   1899.8
new-order         61    59     2    1969.2   3202.0   3717.4
order-status       5     5     0     445.0    667.3    667.3
stock-level        7     7     0     132.9    310.6    310.6
delivery           8     8     0   20179.7  20485.0  20485.0
```

- **tpmC-equivalent** mirrors real TPC-C's actual throughput metric: *successful New-Order transactions per minute* specifically, not total transaction count.
- **Per-type latency** is computed over successful attempts only; `err` counts failures separately (see below for why New-Order shows some). p50/p95/p99 use the standard nearest-rank definition (`ceil((p/100) * n) - 1`) — an earlier version (`floor` instead of `ceil - 1`) overstated small/boundary samples, e.g. reporting the single max value as p95 for exactly 20 samples (Codex review P3 fix, round 10).
- **Delivery is expected to be by far the slowest transaction type** — it's not a bug. One Delivery "transaction" here processes every district in the warehouse in one call (matching the real spec's batch-style semantics), each district potentially doing several index-query round trips plus a commit plus per-line updates. The official TPC-C spec itself acknowledges Delivery is the heavy one: it's the one transaction type the spec explicitly permits to run deferred/asynchronously rather than interactively, for exactly this reason.
- **A small New-Order `TX_ABORTED` rate is expected, not a bug**: every New-Order transaction's first step (incrementing the district's `d_next_o_id`) contends on that district's single row — the same well-known "hot row" property real TPC-C New-Order has. At this benchmark's default scale (2 warehouses × 10 districts = 20 contend-able rows) under concurrency 10, expect roughly low-single-digit percent of New-Order attempts to abort and need a retry (this benchmark does not auto-retry — a caller of this benchmark, or a real application, could choose to). This rate drops further with more warehouses (more districts spreads contention out) and rises with higher concurrency relative to warehouse count — if you see a much higher rate, check your `--warehouses`/`--concurrency` ratio before assuming something's broken.

## Tenancy model

**One tenant per warehouse** (`tpcc-w0001`, `tpcc-w0002`, ...). This is a CloudflareShard-specific design choice, not part of the TPC-C spec: it makes every warehouse's tables naturally isolated (a `table-scan` or `index-query` scoped to one warehouse's tenant only ever sees that warehouse's own rows, with no extra filtering needed), and it's the only way this benchmark can use CloudflareShard's actual tenant-scoped read primitives at all.

## Deviations from official TPC-C

These are deliberate, not oversights — each is a direct consequence of CloudflareShard's real constraints (documented in code comments at the point of each deviation, in `src/transactions.mjs` and `src/generate.mjs`):

1. **Order-Status and Payment's by-customer-*name* lookup variants are dropped; both are ID-only.** The official spec's by-name variant needs a substring/text-search capability CloudflareShard deliberately doesn't have (its only query primitives are exact-tuple `index-query`, cursor-paginated `table-scan`, and mechanically-constructed `mutate` — see the main repo README's "Tenant authorization" section for why raw SQL search isn't offered to tenants).
2. **Stock-Level's server-side aggregate is replaced with `table-scan` + client-side counting.** No aggregation pushdown exists in CloudflareShard at all (this is the same architectural reason TPC-DS was ruled out for this project entirely — see the main issue). Getting the actual scoping right took two review rounds: an early version table-scanned the *whole warehouse's* stock table regardless of district or recent-order scoping (round 1 fix: switch to `index-query` on the district, `.slice(-20)` for "last 20 orders"). That in turn was wrong too (round 10 fix): `index-query` has no cursor and hard-caps at 100 rows (`MAX_INDEX_QUERY_LIMIT`), always returning the *first* 100 matches in ascending order, never the last — so `.slice(-20)` was permanently stuck on the district's 100 earliest orders once `--seed-orders` (or organic growth) pushed past that cap, never seeing a single benchmark-created order. Fixed by walking `table-scan` (which *does* support a cursor) directly against `tpcc_orders`: every order key is `o-{w}-{d}-{o_id}`, zero-padded, so one district's orders form a single contiguous ascending block — the walk only needs to reach the end of district `d`'s own block (not the whole table) to find its true last 20.
3. **New-Order's atomicity is split, not one single all-encompassing transaction.** `/v1/tx` caps a single call at 8 distinct rows (`MAX_TX_PARTICIPANT_KEYS` in `src/index.ts`); a New-Order with the spec's default 5-15 order lines could touch 30+ distinct rows if attempted as one transaction. Instead: one small transaction covers the order header (district update + order insert + new-order insert, 3 rows), and each order line's insert + its stock-row update is its OWN 2-row transaction. Per-line atomicity is preserved (an order line and its stock decrement can never land as only one of the two); whole-order atomicity across all lines is not.
4. **Remote (cross-warehouse) order lines fall back to two independent, non-atomic writes.** `/v1/tx` requires every mutation in one call to share the same tenant; combined with the one-tenant-per-warehouse model, an order line supplied from a different warehouse can never be one atomic transaction (its `order_line` row and the supply warehouse's `stock` row belong to two different tenants). This affects ~1% of order lines, matching the official spec's own cross-warehouse rate.
5. **Reduced default cardinalities** (see the `seed` options table above) for faster demo seeding — scale them up via CLI flags if you want numbers closer to the official spec's.
6. **Immutable reference data (item name/price) is cached client-side after seeding, rather than re-read from the server on every transaction.** The official spec mandates fresh reads of these fields on every transaction; since they never change post-load, caching them is a deliberate simplification that meaningfully reduces round trips without affecting correctness of the actual transaction semantics being measured (writes to `district`/`warehouse`/`customer`'s *mutable* fields — tax-adjacent YTD counters, balances — are always read fresh, since those genuinely do change).
7. **The item catalog is duplicated per warehouse-tenant rather than shared globally.** Real TPC-C has one global item catalog across all warehouses; CloudflareShard's tenant-scoped read primitives have no cross-tenant read path, so each warehouse-tenant gets its own identical copy instead.
8. **Pre-existing orders at seed time are opt-in (`--seed-orders`), not default-on** — see the `seed` options table above for why.
9. **Every counter update (stock quantity/YTD, warehouse/district YTD, customer balance/payment/delivery counts) goes through a `/v1/mutate` call with a compare-and-swap `where` guard and a bounded read-compute-write retry loop, never bundled inside a `/v1/tx`.** CloudflareShard's structured mutations only support SETting a column to a literal client-supplied value — there's no server-side arithmetic UPDATE (`col = col - ?`) — so every increment/decrement here is computed client-side from a prior read, then written back, and needs a guard against a concurrent transaction reading the same stale value. Bundling that guard inside a `/v1/tx` (an earlier version of this benchmark did, for New-Order's stock update and Payment/Delivery's counters) makes a losing guard *invisible*: `/v1/tx` reports the whole transaction "committed" with no per-mutation `rowsAffected`, so the benchmark could count a transaction as fully successful even when its counter update silently didn't apply. Only `/v1/mutate`'s per-call `rowsAffected` makes a real, safe retry possible — this is why New-Order's stock update, Payment's warehouse/district/customer updates, and Delivery's customer credit all ended up as separate `/v1/mutate` calls with their own retry loop instead of one atomic multi-row transaction, a real and deliberate trade-off (see item 10 for what that cost New-Order specifically, and `src/transactions.mjs`'s file-header comment for the full history — five rounds of review found and closed real gaps here, each verified live under deliberately extreme contention). A sixth gap in this same pattern, common to every one of these retry loops, was found in round 13: each attempt computes its delta from a FRESH read, so a clean `rowsAffected: 0` response is safe to treat as "try again" (the next attempt's fresh read picks up whatever's actually true) — but if the `mutate()` call itself THROWS ambiguously (an ordinary network/DO-level blip, not a clean HTTP response), naively moving to the next attempt is unsafe: that attempt's fresh read might show the ambiguous one's change already applied, and computing a new delta on top of it would apply the SAME intended change a second time. Fixed with `mutateResolvingAmbiguity`, a small wrapper every one of these calls now goes through: on an ambiguous (non-`ApiError`) throw, it retries the IDENTICAL call (same values/where/`requestId`, no re-read, bounded at `AMBIGUITY_RESOLUTION_ATTEMPTS`) rather than guessing — `ShardDO`'s `applied_requests` table (`src/shard.ts`) keys on `requestId` and stores a hash of the actual sql/params, so replaying the same call either returns the ORIGINAL attempt's true, cached outcome (if it actually applied) or genuinely executes for the first time (if it never did), without ever risking an independently-computed second delta landing on an unknown prior state. Applied to all five of these call sites (New-Order's stock update, Payment's three counters, Delivery's customer credit) for consistency, though the severity differs: Payment/New-Order's stock have no retry mechanism that revisits the same logical change, so an unresolved ambiguity there would at worst strand one small, undetected delta; Delivery's credit sits inside the marker-restore-and-retry mechanism (see item 11), where an unresolved ambiguity could double-credit the same order on a later Delivery pass — the more severe of the two, and the one Codex flagged directly alongside New-Order's stock case.
10. **New-Order gave up whole-order atomicity entirely, in exchange for every step being individually safe, detectable, and (where it matters) compensated on failure.** The header (district update + order insert) is one small 2-row `/v1/tx`. Each line's `order_line` insert and stock update are two separate `/v1/mutate` calls (see item 9) — if the stock update can't apply after retrying, the already-inserted `order_line` row is deleted immediately. The `new_order` marker (what makes an order visible to Delivery at all) is inserted last, only once every line has genuinely committed, and if any line ultimately fails, `compensateFailedOrder` reverses every already-committed line (deletes its `order_line` row, reverts its stock update via its own compare-and-swap) and deletes the orphaned `orders` row (`d_next_o_id` is deliberately not decremented back — a gap in the o_id sequence is expected and harmless, like a rolled-back insert in a real database's auto-increment). Verified live: a high-concurrency, small-item-pool stress run against fresh warehouses repeatedly triggered both the retry loops and the compensation path; cross-referencing every order/order_line/new_order row server-side afterward found zero orphaned and zero incomplete-line-count orders. Full history of the five specific bugs this converged from is in `src/transactions.mjs`'s file-header comment. A sixth was found in round 10: the final `new_order` marker insert itself (not any line, and not the header) could fail on a transient error *after* every line and the header had already committed — with nothing left to compensate, but the order still permanently invisible to Delivery without its marker. Fixed by retrying that one insert (bounded, `MUTATION_RETRY_ATTEMPTS`) under a single stable `requestId` reused across attempts — safe to retry blindly, unlike a counter update, because `/v1/mutate`'s underlying dedupe cache (`applied_requests` in `src/shard.ts`) makes a retry of an already-applied insert replay its cached outcome instead of erroring or double-inserting. If every attempt still fails, this throws rather than guessing at compensation: there's no way to tell "never applied" apart from "applied once, response lost every retry since," and reverting a possibly-already-marked order would risk corrupting it instead. A seventh (round 11): the header tx's district update (`d_next_o_id`) was an unconditional SET with no guard — a request that read `d_next_o_id` and then stalled could still commit later than a faster order that had since advanced the same counter, blindly setting it back down and handing out an already-used `o_id` again. Fixed by adding a `where: { d_next_o_id: o_id }` guard to that one participant; per this same file's established `/v1/tx` semantics, a guard miss just means that participant has no effect while the order insert alongside it still succeeds or fails independently (a duplicate-key failure there is self-resolving, recorded as an ordinary failed attempt) — no retry loop needed, since there's never a legitimate reason to retry a stale header. An eighth (round 12, the same class of bug as the sixth above but for the OTHER end of this function): the header `/v1/tx` call itself could throw ambiguously on a transient network error, exiting `newOrder` before line processing or compensation ever ran and leaving a genuinely committed `orders` row with no lines and no marker. Fixed the same way as the sixth: retry (bounded, `MUTATION_RETRY_ATTEMPTS`) under a single stable `requestId`, safe because `/v1/tx`'s `txId` is derived from `(tenantId, requestId)` (`src/coordinator.ts`'s `handleBegin`) — replaying an already-committed header returns its cached outcome rather than re-executing. This must NOT retry on a clean, definitive error response though (a real `409 TX_ABORTED` from genuine prepare-phase lock contention means the coordinator has already recorded that `txId` as aborted for good — retrying with the same `requestId` would just replay the same aborted response every time, burning the whole retry budget instead of surfacing the real failure immediately, as this benchmark already does elsewhere); only a thrown error that isn't a clean `ApiError` (a network-level failure with no definitive server response at all) triggers a retry. A ninth (also round 12): a line's `order_line` insert itself could throw the same ambiguous way, and used to run BEFORE the try block that contains its own cleanup logic — so an ambiguous insert failure skipped that cleanup entirely, potentially orphaning an inserted row under a header that goes on to get deleted. Fixed by moving the insert inside the same try block, so it now shares the exact delete-then-rethrow cleanup the stock-update failure path already used. A tenth (round 14): `compensateFailedOrder` reversed already-committed lines in forward (insertion) order, which is only safe when every line touches a DISTINCT stock row. When `--items` is smaller than a line count (or the within-one-order dedup loop's bounded retry still collides), two lines in the SAME order can reference the SAME `(supplyWarehouseId, i_id)` stock row, one applying its decrement on top of the other's -- reversing the EARLIER of the two first tries a compare-and-swap whose `where` no longer matches (the later line already changed the row again), safely no-ops, and is never retried, permanently stranding that decrement even though the whole order is being compensated. Fixed by reversing in LIFO order instead: `succeededLines.push()` happens as each line's own retry loop resolves, so for two lines racing the same row, push order already reflects the TRUE server-side application order (whichever won the CAS first is pushed first, the other necessarily read and built on top of it) -- reversing LIFO always unwinds a same-row chain in the exact reverse of how it was built, so each reversal's `where` matches the state the reversal just before it left behind, fully undoing both instead of stranding one.
11. **Delivery claims a district's oldest undelivered order via its own checked `/v1/mutate` delete before touching anything else**, so a concurrent Delivery worker that read the same "oldest" order (before the first worker's claim committed) sees `rowsAffected: 0` on its own claim attempt and correctly skips the order — rather than re-crediting the customer for an order someone else already delivered. If the marker is claimed but anything afterward fails *before* the customer credit succeeds, the marker is restored so a later pass can retry the whole order; once the credit is confirmed, the marker is never restored again (a failure marking individual lines' delivery dates afterward must not cause a re-credit). This shape converged from five real bugs found by review — the original `/v1/tx`-bundled marker-delete-plus-credit having the exact same invisible-guard problem item 9 describes, a stranding gap after the claim-fence fix, a double-credit regression from an earlier fix's try/catch scope being too wide, a genuine P1: CloudflareShard's secondary-index maintenance for `/v1/mutate` is dispatched via `ctx.waitUntil()` *after* the base row write already returned to the caller — so Delivery's `idx_order_line_by_order` lookup could run before every one of a just-placed order's lines had a queryable index entry yet, summing an incomplete set, crediting the customer too little, and then deleting the marker — permanently losing whichever lines hadn't caught up (no marker survives for a later pass to find them by; fixed by waiting, bounded retry, for the index to report at least `orders.o_ol_cnt` lines — a base-row field written synchronously, so always trustworthy as the target count — before trusting the result) — and, round 11, the claim delete itself could throw on a transient error with no way to tell whether it had actually committed before the error, which would skip past every safety net below it (the try/catch that restores the marker on a *later* failure never even starts). Fixed by wrapping the claim call itself: on any such error, best-effort reinsert the marker (a no-op duplicate if the delete never actually applied, a real restoration if it did) before rethrowing. Order-Status reads via the identical `idx_order_line_by_order` index and has the identical underlying lag, but deliberately doesn't retry for it: a moment's staleness there just means reporting an order that's a few hundred milliseconds old, self-healing on the next call, not the permanent data loss Delivery's marker deletion would turn it into.
12. **Payment draws a remote (different-warehouse) customer ~15% of the time whenever more than one warehouse is seeded, matching the official spec's cross-warehouse rate** (Codex review P2 fix, round 11 — an earlier version always resolved the customer against the terminal's own warehouse, silently dropping this path and the cross-warehouse load it's meant to exercise entirely). warehouse/district YTD always update on the terminal's own (home) warehouse regardless; only the customer lookup/update resolves against the remote warehouse's tenant instead — straightforward specifically because Payment's three counters were already three independent `/v1/mutate` calls (see item 9), never one bundled `/v1/tx` that a same-tenant restriction would have made incompatible with a remote customer anyway.

13. **Two small usability gaps in the benchmark's own tooling, not its transaction logic** (Codex review P3 fixes, round 12): the exported `seed()` helper's `seedOrders` parameter defaulted to `true`, contradicting the documented "opt-in, off by default" behavior — harmless via the CLI (`src/run.mjs` always passes this flag explicitly), but a script importing `seed()` directly and omitting the option silently got the expensive path. Fixed by matching the default to what's documented (`false`). Separately, a base URL with a trailing slash (e.g. `http://localhost:8787/`, an easy thing to paste) concatenated into a double-slashed path (`//admin/create-table`), which 404'd against every one of CloudflareShard's exact-pathname routes despite the host being perfectly correct. Fixed by trimming trailing slashes before joining.

None of this changes what the benchmark actually demonstrates: real `CoordinatorDO` 2PC transactions, real sharded `mutate` writes, real `index-query`/`table-scan` reads, under a realistic (if not officially-compliant) OLTP transaction mix and cardinality.
